## 1. Описание задачи

Необходимо разработать функционал для комплексного отслеживания взаимодействий между резидентом и структурой фонда в экосистеме Stellar. Модуль должен обеспечивать полную прозрачность движений активов, операций и структурных связей.

---

## 2. Требуемые сценарии

### 2.1 Мониторинг аккаунта эмитента

**Цель:** Получение полной информации о движениях по аккаунту эмитента.

**Требования:**
- Отслеживание всех входящих и исходящих транзакций
- Фильтрация по типам операций (payments, trades, swaps)
- Временная шкала операций с возможностью детализации
- Агрегирование данных по периодам (день, неделя, месяц)

**Выходные данные:**
- Список транзакций с указанием контрагентов
- Суммы и типы активов
- Метки времени операций
- Ссылки на оригинальные транзакции в Horizon

---

### 2.2 Структура аккаунтов и ассетов эмитента

**Цель:** Визуализация и анализ структуры владения активами эмитента.

**Требования:**
- Карта связей между аккаунтами (граф зависимостей)
- Иерархия владения активами
- Список всех trustlines эмитента
- Анализ распределения активов по контрагентам
- Отслеживание создания и удаления trustlines

**Выходные данные:**
- Древовидная/графовая структура связей
- Сводка по типам активов (токены, стейблкоины)
- Статистика по объемам доверительных линий

---

### 2.3 Хронология взаимодействий с резидентом

**Цель:** Определение момента начала и истории взаимодействия структуры фонда с резидентом.

**Требования:**
- Первая транзакция между структурой фонда и резидентом
- Полная история операций с резидентом
- Фильтрация по типам операций
- Временная шкала с возможностью масштабирования
- Определение "точки входа" резидента в структуру

**Выходные данные:**
- Timeline взаимодействий
- Список первых контактов
- Статистика активности по периодам

---

### 2.4 Операции с токенами и стейблкоинами

**Цель:** Детальный анализ операций с различными типами активов.

**Требования:**
- Классификация операций по типам активов
- Отслеживание swap-операций между токенами
- Мониторинг операций со стейблкоинами (USDC, EURC и др.)
- Анализ объемов конвертации
- Выявление подозрительных паттернов

**Выходные данные:**
- Разбивка по типам активов
- Курсы конвертации (если применимо)
- Объемы операций в разрезе активов

---

### 2.5 Обменные операции

**Цель:** Полная картина обменных операций между структурой фонда и резидентом.

**Требования:**
- Отслеживание всех trade-операций
- Анализ условий обмена (спреды, комиссии)
- Сопоставление с orderbook-данными
- Выявление отклонений от рыночных условий
- История обменных пар

**Выходные данные:**
- Список обменов с детализацией
- Анализ рыночных условий
- Расхождения с рыночными показателями

---

### 2.6 Отслеживание движения учетных токенов эмитента

**Цель:** Анализ распространения учетных токенов эмитента в структуре фонда.

**Требования:**
- Карта движения токенов от эмитента
- Отслеживание промежуточных держателей
- Анализ цепочек трансферов
- Выявление концентрации токенов
- Мониторинг "размытия" токенов по структуре

**Выходные данные:**
- Граф движения токенов
- Список всех промежуточных держателей
- Анализ пулов токенов

## 4. Технические требования

### 4.1 Интеграция с существующей системой

Модуль должен:
- Использовать существующую инфраструктуру stellar-scrypt
- Работать с Google Apps Script (GAS)
- Интегрироваться с Google Sheets для вывода данных
- Использовать Horizon API для получения данных
- Следовать паттернам проектирования из [`systemPatterns.md`](memory-bank/systemPatterns.md)

**Существующие точки интеграции:**
- Лист `CONST`: добавление новых конфигурационных ключей (см. [ADMIN_SETUP_AND_CONFIG.md](ADMIN_SETUP_AND_CONFIG.md))
- Лист `TRANSFERS`: существующие данные о переводах для анализа
- Лист `RESIDENTS`: информация о резидентах и их аккаунтах
- Лист `PROJECT_MAP`: маппинг project_id для связывания данных
- Функции [`parseConstSheet()`](clasp/Резиденты%20Мабиз.js:700), [`parseResidentsSheet()`](clasp/Резиденты%20Мабиз.js:731) для чтения данных
- Функции [`writeDebugLog()`](clasp/Резиденты%20Мабиз.js:1307) для логирования

**Новые листы (предлагаемые):**
- `RESIDENT_TRACKING`: основные данные отслеживания
- `ISSUER_STRUCTURE`: структура аккаунтов эмитента
- `TOKEN_FLOWS`: движение токенов по структуре фонда
- `RESIDENT_TIMELINE`: хронология взаимодействий

### 4.2 Производительность

- Обработка больших объемов транзакций (учесть лимиты GAS)
- Кэширование результатов через [`CacheService`](clasp/Резиденты%20Мабиз.js:645)
- Инкрементальное обновление данных (курсоры как в [`syncStellarTransfers()`](clasp/Резиденты%20Мабиз.js:149))
- Учёт лимитов Horizon API rate limiting

### 4.3 UI/UX

- Новые пункты меню `Stellar` через [`ui.createMenu()`](clasp/Резиденты%20Мабиз.js:39)
- Визуализация графов и временных шкал (возможно, через встроенные диаграммы Sheets)
- Фильтрация и поиск по данным (по аналогии с [`showTransactionsBetweenAddresses()`](clasp/Резиденты%20Мабиз.js:1978))
- Экспорт данных для дальнейшего анализа
- Следовать стилю существующих диалогов и уведомлений
