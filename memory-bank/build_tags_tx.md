## ТЗ: `build_tags_tx` — генератор XDR для Stellar manage_data (CLI 23.4.1)

### 0) Цель

Сделать zsh-скрипт, который:

1. Берёт два списка Stellar-адресов из отдельных конфигов:

* `p2.txt` — список **operation source accounts** (источники тегов)
* `p3.txt` — список **целевых адресов** (значения тегов)

2. Генерирует **одну транзакцию** Stellar (PUBLIC сеть), содержащую **N2 × N3** операций `manage_data`.

3. В каждой операции:

* `data_name` = **тег** вида `D<i>_<j>` (или иной согласованный формат), где:

  * `i` — индекс источника из `p2` (1..N2)
  * `j` — индекс цели из `p3` (1..N3)
  * гарантировать уникальность имени тега внутри аккаунта-источника
* `data_value` = **Stellar-адрес цели** (из `p3`) в виде **hex ASCII** (как требует CLI)

4. Для каждой операции `op_source` = соответствующий адрес из `p2` (i-й источник).
   При этом учесть ограничения `stellar-cli 23.4.1`:

* `stellar tx new manage-data` **не принимает** `--operation-source-account`
* `stellar tx new manage-data` **не принимает** `--inclusion-fee`
* `--network` нельзя ставить на уровень `tx new`, сеть указывается на уровне подкоманд (или через passphrase/rpc-url)
* fee задаётся через `stellar fees use --amount ...` (или иным способом, подтверждённым в справке текущей версии)

5. Скрипт пишет:

* `tx.xdr` — итоговый base64 XDR
* `tx.debug.json` — decode транзакции (для проверок)
* `out/run_YYYYMMDD_HHMMSS.log` — лог выполнения

---

### 1) Окружение и ограничения

* ОС: Manjaro / Linux
* Shell: **zsh** (скрипт должен запускаться `zsh build_tags_tx.zsh`)
* CLI: `stellar --version` = **23.4.1**
* Internet: есть доступ к RPC endpoint (pubnet)
* Ноды нет и не нужна
* Скрипт нельзя делать “всё в одну команду”, нужно идти через корректные subcommands `stellar tx new`, `stellar tx operation add`, `stellar tx update`, `stellar tx decode`.

---

### 2) Входные данные

Файлы (по умолчанию рядом со скриптом, но должны быть флаги на кастомные пути):

* `p2.txt` — по одному G… на строку
* `p3.txt` — по одному G… на строку (возможны дубли — **не удалять**, обрабатывать как есть)
* (опционально) `config.env` или переменные окружения для RPC/Network passphrase/fee и т.д.

Параметры транзакции:

* Network passphrase: `"Public Global Stellar Network ; September 2015"`
* RPC URL: по умолчанию `https://stellar-soroban-public.nodies.app` (параметризуемо)
* Tx source: по умолчанию = **третий адрес из p2** (если есть), иначе первый (параметризуемо)
* Sequence: режим по умолчанию — `stellar tx update seq-num next` (самый надёжный).
  Опционально: режим `--seq-manual <number>` через правку JSON (если реально возможно); если невозможно — честно выключить и объяснить.

Fee:

* Скрипт должен посчитать `min_fee = base_fee * (N2*N3)` при base_fee=100.
* Перед сборкой выставить `stellar fees use --amount <fee>` где `<fee>` = max(min_fee, заданного пользователем).
* Скрипт должен вывести в лог: `ops_count`, `min_fee`, `configured_fee`.

---

### 3) Функциональные требования

#### 3.1 Сборка транзакции

Алгоритм сборки должен быть совместим с CLI 23.4.1:

1. Установить fee через `stellar fees use --amount ...` (или альтернативно, но только если подтверждено `--help` текущей версии).
2. Создать транзакцию **первой** операцией через `stellar tx new manage-data`:

   * `--source-account` = `TX_SOURCE` (это и tx source, и source первой операции)
   * `--data-name` = `D1_1` (или по согласованной формуле, но индекс должен соответствовать i/j)
   * `--data-value` = hex(первого адреса из p3)
   * `--rpc-url`, `--network-passphrase`
   * `--build-only`
3. Добавить остальные операции через `stellar tx operation add manage-data`:

   * stdin = текущий XDR
   * `--source-account` = `TX_SOURCE` (tx source)
   * `--operation-source-account` = текущий `SRC` из `p2`
   * `--data-name`, `--data-value`
   * `--rpc-url`, `--network-passphrase`
   * `--build-only`
4. Обновить sequence: `stellar tx update seq-num next ...` на готовом XDR.
5. Сохранить итог в `tx.xdr`.
6. Сгенерировать `tx.debug.json` через `stellar tx decode`.

Важно: учесть, что первая операция будет иметь source = `TX_SOURCE`.
Но требования пользователя: “для каждого SRC из p2” — значит первая операция должна соответствовать i и j так, чтобы не было несостыковки.
**Правило привязки**:

* Считаем `TX_SOURCE` = `p2[3]` (как сейчас), и это один из источников — ок.
* Первая операция должна быть именно для `i = index(TX_SOURCE)` и `j=1`.
  Если `TX_SOURCE` не входит в p2, скрипт должен упасть с понятной ошибкой.

#### 3.2 HEX кодирование

`data_value` в CLI требует hex-строку.
Скрипт обязан:

* конвертировать строку вида `G...` в hex ASCII без пробелов
* валидировать длину ≤ 64 bytes (то есть ≤ 128 hex chars); Stellar-адрес 56 символов => 56 bytes => ок.
* иметь отдельную функцию `to_hex_ascii()`

#### 3.3 Валидации и диагностика

Скрипт должен:

* проверять, что все строки p2/p3 начинаются с `G` и длина 56 (приблизительная валидация)
* выводить в лог:

  * количество источников и целей
  * количество ops
  * какой TX_SOURCE
  * какой RPC_URL и passphrase (passphrase можно скрыть)
  * пример 1–2 первых операций (name/value/source)
* после сборки: декодировать и проверить:

  * `operations.length == N2*N3`
  * `sourceAccount == TX_SOURCE`
  * для каждой операции:

    * name соответствует формуле
    * value hex соответствует исходному адресу (декодировать обратно для проверки)
    * sourceAccount операции соответствует ожидаемому SRC
* при несоответствии — завершать с ошибкой и указывать первую найденную проблему.

### 4) Выходы

* `tx.xdr` — base64 XDR транзакции
* `tx.debug.json` — JSON decode
* `out/run_*.log` — лог
* (опционально) `tx.summary.txt` — краткое резюме: ops_count, fee, tx_source, список требуемых подписей (p2)

---

### 5) Тест-план (обязателен)

Оркестратор должен **автоматически прогнать**:

1. Smoke test:

* p2=3 строки, p3=9 строк → ops=27
* скрипт завершается успешно
* `stellar tx decode` показывает 27 операций

2. Дубли в p3:

* вставить дубль адреса → ops всё равно = N2*N3
* имена тегов уникальны, конфликтов нет

3. Неверный адрес:

* один адрес не 56 символов → скрипт падает до сборки, пишет понятную ошибку

4. Fee:

* вычисленный `min_fee` корректен
* в decode видно fee ≥ min_fee (или подтверждённый параметр, как это видно в JSON)

5. Стабильность:

* повторный запуск перезаписывает `tx.xdr` и создаёт новый `run_*.log`

---

### 6) Что запрещено

* Нельзя требовать поднятия ноды
* Нельзя использовать неподтверждённые флаги (`--network` на неправильном уровне, `--inclusion-fee` там, где его нет)
* Нельзя “на глаз” генерировать XDR строкой
* Нельзя менять смысл: p2/p3 должны быть внешними файлами, а теги — name, адрес — value.

---

### 7) Критерий готовности (Definition of Done)

* Скрипт запускается одной командой `zsh build_tags_tx.zsh`
* Создаёт XDR и debug JSON
* Встроенная проверка подтверждает 27 ops и корректное соответствие источников/тегов/значений
* Тест-план выполнен, логи приложены
